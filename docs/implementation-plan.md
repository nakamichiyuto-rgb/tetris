# テトリスWEBアプリ 実装計画書

## 1. 目的

本計画書は、[要件定義書](/Users/soyoo/Documents/Repo/Tetris/docs/tetris-requirements.md) をもとに、ローカルで動作するテトリスWEBアプリを効率よく実装するための進め方を整理したものである。
まずは最低限遊べるMVPを完成させ、その後に操作性や見た目、拡張機能を段階的に追加する。

## 2. 開発方針

- 最初にMVPを完成させる
- ゲームロジックと描画処理を分離する
- ローカル実行しやすいシンプルな構成を優先する
- 追加機能を後から載せやすい設計にする
- 先に正しく動く状態を作り、その後にUIと演出を磨く

## 3. 技術方針

### 3.1 推奨構成

- フロントエンド: HTML / CSS / JavaScript
- 開発方式: ローカル起動可能な軽量構成
- 描画方式: `canvas` または DOM グリッド
- 推奨: 初期実装は `canvas` ベース

### 3.2 推奨理由

- テトリスは盤面描画の更新頻度が高く、`canvas` の相性が良い
- 単一ページ構成で完結しやすい
- ロジックを独立させやすく、将来的な TypeScript 化もしやすい

## 8. MVP定義

MVPでは以下を満たせば実装完了とする。

- 盤面が表示される
- 7種ブロックが出現する
- 左右移動、回転、ソフトドロップ、ハードドロップができる
- ブロック固定とライン消去ができる
- スコアが加算される
- レベルが上がる
- 次ブロックが表示される
- 一時停止とリスタートができる
- ゲームオーバー表示が出る

## 9. リスクと対策

### リスク1: 回転処理が複雑になる

対策:
- 最初はシンプルな回転ロジックで実装する
- 壁際の高度な回転補正は拡張扱いにする

### リスク2: ロジックと描画が密結合になる

対策:
- 盤面状態と描画処理を別ファイルに分ける
- 描画は状態を受け取って反映するだけにする

### リスク3: 入力と自動落下の競合で挙動が不安定になる

対策:
- 状態更新の入口を `game.js` に集約する
- 1フレーム内の更新順序を固定する

## 10. 受け入れチェック計画

- 起動して即プレイ可能か
- 盤面外へブロックが移動しないか
- ブロックが他ブロックを貫通しないか
- ライン完成時に正しく消えるか
- スコアとレベルが期待通りに更新されるか
- 一時停止中に自動落下しないか
- リスタート後に状態が完全初期化されるか

## 11. 実装着手順
